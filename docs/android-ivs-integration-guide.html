Android ↔ IVS Integration Guide

Everything you need to integrate a native Android application with GBG's Identity Verification Suite — authentication, API contract, error handling, and polling pattern.

📋

Overview

The GBG Identity Verification Suite (IVS) is the engine-agnostic command-and-control layer that sits between your Android app and the underlying document verification engines. Your app never calls verification engines directly — all communication goes through IVS.

This guide covers the complete four-step integration. Step 3 (Submit Images) uses a new endpoint built specifically for the Android integration — all other steps reuse existing IVS infrastructure.

Step What Happens Who Does It Status
1 — Create Session Initialize a verification session in IVS App → IVS Existing
2 — Capture Images Capture document photos on-device App + SDK only SDK Only  Docs ↗
3 — Submit Images Send captured images to IVS for processing App → IVS New Endpoint
4 — Poll for Status Check verification status until complete App → IVS Existing

Base URL:

https://ditto.gbg.com

All paths in this guide are relative to this base URL.

🔐

Authentication

All IVS API calls use API key authentication. You will be provided an API key for the Enterprise Rent-A-Car demo environment. Include it in every request header as follows:

Authorization: Bearer <your_ivs_api_key>

Every request in Steps 1, 3, and 4 must include this header. There is no session token or Firebase auth — the API key is your sole credential for all calls.

Rate Limit: 30 requests per minute. Your API key is configured with a maximum of 30 requests per minute across all endpoints combined. This is well above what a normal verification flow requires — you will only approach this limit if running automated tests in rapid succession. If exceeded, IVS will return 429 Too Many Requests. Back off and retry after a short delay.

⚠️ Security Requirement

Store the API key in Android Keystore. Do not hardcode it in source, commit it to version control, or store it in SharedPreferences. Treat it as you would any production secret.

1

Create Session

Initialize a verification session before capturing any images. The session ID returned here is your reference for all subsequent steps.

POST /api/verification/sessions
Authorization: Bearer <ivs_api_key>  |  Content-Type: application/json
Request Body
{ "verificationType": "docAuth", "returnUrl": "gbgdemo://result", "faceMatchEnabled": false, "sensorType": "Mobile", "referenceId": "optional-your-reference" }
FieldTypeNotes
verificationType string Always "docAuth" for document authentication flows
sensorType string Always hardcode "Mobile". Do not make this configurable — it maps to the correct processing mode in the verification engine
faceMatchEnabled boolean Always hardcode false. Do not implement face match functionality at this time
returnUrl string Use "gbgdemo://result". This is a web-only concept — include it but the app can discard the verifyUrl in the response
referenceId string Optional. Use for your own internal tracking if needed
Response — 201 Created
{ "sessionId": "vs_abc123...", "verifyUrl": "https://...", // web-only — discard "expiresAt": "2026-04-16T14:30:00Z" }

Store the sessionId only. The verifyUrl is a web-only concept and can be discarded. The expiresAt timestamp may be useful for proactive expiry detection (see below).

⚠️ Session Expiry — 30 Minutes

Sessions expire 30 minutes after creation. If the user takes longer than 30 minutes between session creation and image submission — for example, if the app is backgrounded — the session will be expired and cannot accept images. When this happens, your app should silently create a new session and resume the flow from Step 1. Do not surface the internal expiry detail to the user; a "starting a fresh verification" message is sufficient.

2

Capture Images

This step is entirely on-device. IVS is not involved.

Use the GBG Smart Capture Android SDK (provided separately as a .aar file) to capture front and back images of the identity document. The SDK handles camera configuration, lighting guidance, and image quality checks automatically. For this integration you will use the SmartCapture Document module, which provides auto-capture and user guidance for identity documents.

📖 SDK Documentation: smartcapture-docs.idscan.cloud/docs/android/intro — covers all available modules including Document Camera, Face Camera, and NFC Chip Authentication.

⚠️ Do Not Recompress Images

Use the SDK's image output directly. Do not recompress or transcode images after the SDK returns them. Recompression degrades quality in ways that can affect verification accuracy. Pass SDK output to Step 3 as-is.

3

Submit Images

Send the captured images to IVS for processing. IVS returns 202 Accepted immediately and processes the images asynchronously — proceed to polling after receiving the acknowledgement.

🚧 New Capability — Coming Shortly

This endpoint is a new capability being built specifically for this Android integration. It does not yet exist in the environment but will be available shortly. You will be notified when it is ready. In the meantime, you can proceed with Steps 1, 2, and 4 — Step 4 can be validated independently once a test session is provided alongside your API key.

POST /api/verification/sessions/{sessionId}/images
Authorization: Bearer <ivs_api_key>  |  Content-Type: application/json
Request Body
{ "frontImage": "data:image/jpeg;base64,/9j/...", "backImage": "data:image/jpeg;base64,/9j/..." }

Image encoding: Base64-encoded with the data URI prefix (data:image/jpeg;base64,). Encode the SDK output directly — no additional processing. Both frontImage and backImage are required. Do not implement face match at this time.

Response — 202 Accepted
{ "sessionId": "vs_abc123...", "status": "processing" }
⚠️ Submit-Once Constraint

A session can only receive one image submission. A second submit to the same sessionId will return 409 Conflict. If submission fails for network reasons, retry the same request — do not create a new session.

4

Poll for Status

Begin polling immediately after receiving the 202 from Step 3. Continue until you receive a terminal status or exhaust your retry budget.

🧪 Validating Step 4 independently: You don't need Step 3 to be complete to validate your polling implementation. Once your API key is issued, a completed test session ID will be provided that you can query directly. Use it to confirm your GET call, response parsing, and result display are all working correctly before the image submission capability is available.

GET /api/verification/sessions/{sessionId}
Authorization: Bearer <ivs_api_key>
Response — 200 OK
{ "sessionId": "vs_abc123...", "status": "pending | processing | completed | failed | error | timed_out", "result": { "ivsOverallResult": "pass | fail | warn | error", "decision": "Pass | Fail | Attention", "documentVerified": true, "documentExpired": false, "attentionNotices": [], "failureReason": null }, "createdAt": "2026-04-16T14:00:00Z", "completedAt": "2026-04-16T14:00:18Z" }

Polling pattern: Poll every 4 seconds, up to 10 attempts (~40 seconds total). Stop polling as soon as status is one of: completed, failed, error, or timed_out.

🔁

Timeout Handling

1

Poll every 4 seconds

Begin immediately after the 202 response from Step 3. Each attempt calls GET /api/verification/sessions/{sessionId}.

2

Stop on terminal status

Stop polling immediately when status is completed, failed, error, or timed_out. Display the result to the user.

3

After 10 attempts — offer retry

If 10 poll attempts complete without a terminal status, display a message: "This is taking longer than expected." Offer the user a Try Again action. This resets to a fresh 10-attempt cycle against the same sessionId — do not create a new session. The verification is still processing server-side.

📊

Status Reference

StatusMeaningAction
pending Session created, awaiting image submission Should not appear during polling
processing Images received, verification in progress Continue polling
completed Verification finished — inspect result Stop polling, display result
failed Verification failed (e.g. unreadable document) Stop polling, show failure
error Unexpected processing error Stop polling, show error
timed_out IVS processing timed out server-side Stop polling, show error

Result Interpretation

ivsOverallResultdecisionMeaning
pass Pass Document verified successfully
warn Attention Verified with notices — inspect attentionNotices for detail
fail Fail Verification failed — inspect failureReason
error — Processing error — treat as a failed verification

Error Reference

HTTP StatusScenarioNotes
400 Bad Request Session expired Session lifetime is 30 minutes. Create a new session and resume from Step 1
400 Bad Request Session not in pending state Session was already submitted or is in the wrong state for this operation
400 Bad Request Missing required image fields Ensure frontImage and backImage are always present
401 Unauthorized Missing or invalid API key Verify the Authorization header is present and correctly formatted: Bearer <key>
404 Not Found Session ID does not exist Do not retry with the same ID — check your session creation logic
409 Conflict Images already submitted to this session Sessions are submit-once. If you need to retry after a network failure, re-submit the same request — do not create a new session. This 409 indicates the original submission succeeded
429 Too Many Requests Rate limit exceeded Your key allows 30 requests per minute. Back off and retry after a short delay — this will only occur during rapid automated testing
500 Internal Server Error Unexpected server error Retry once after a short delay. If persistent, escalate to the GBG team

Constraints Summary

  • sensorType: "Mobile" must be hardcoded in the session creation request — do not make this configurable
  • Images must be base64-encoded with the data URI prefix: data:image/jpeg;base64,...
  • Do not recompress or transcode SDK image output — use it directly
  • Both frontImage and backImage are required — do not implement face match at this time
  • Sessions expire 30 minutes after creation — silently create a new session if this occurs; do not expose the error to the user
  • Sessions are submit-once — a second image submission returns 409; use this as a signal that the original submit succeeded
  • API key must be stored in Android Keystore — not in source code or SharedPreferences
